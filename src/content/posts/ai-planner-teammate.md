## 목차

## 배경

운영 중인 키오스크·관리 플랫폼은 화면이 먼저 있고 기획 문서는 뒤따라오지 못한 상태가 흔하다. QA·기획·신규 입사자가 화면을 이해하려면 '지금 화면이 어떻게 동작하는가'가 문서로 있어야 하는데, 이걸 사람이 수작업으로 만들면 화면 수십 종 분량이라 밀린다.

'읽고 재구성'은 AI가 가장 잘하는 일이다. 그래서 이 일을 맡을 **AI 기획자 팀원**을 만들었다.

## 팀원 설계 — 4층 구조

| 층 | 성격 | 담는 것 |
| --- | --- | --- |
| 페르소나 | 불변 · 누구인가 | 기획자로서의 정체성·관점 |
| 헌법 | 반영구 · 제품 규칙 | 용어·톤·ALWAYS/NEVER 원칙. 상황이 바뀌어도 안 건드림 |
| 방법론 | 재사용 · 산출 계약 | 스킬별 3-파일(규칙+템플릿+예시) — 역기획·디스크립션 작성법 |
| 상태 | 가변 · 매일 변함 | 지금 어디까지 왔나 — 세션이 바뀌어도 여기서 이어감 |

핵심 원리는 **불변의 헌법과 가변의 상태 분리**. 상황이 바뀌면 상태만 갱신하고 기획 원칙은 건드리지 않는다. 세션이 끊겨도 기획 원칙은 보존되고 진행 위치만 갱신하면 된다.

역할 경계도 명확히 그었다 — **기획자≠개발자**. 기획자는 명세까지만 만들고(개발·배포 금지), 구현은 「개발 핸드오프」 블록(무엇/수용기준/영향범위/제약, 구현법은 비움)으로 별도 개발자 세션에 넘긴다.

## 산출물 — 스펙 패널은 이렇게 생겼다

기획자의 방법론 층이 정의하는 산출 규격. 소스에서 화면 구조·전환 흐름·문구키를 추출해 Figma에 화면 + 스펙 패널을 쌍으로 배치하고, 화면 위 번호 뱃지 ①②③…과 스펙 항목을 1:1로 대응시킨다 (도메인 각색 예시):

> **③ 시작하기 버튼**
>
> 화면 중앙의 큰 주황색 버튼. 손님이 이 버튼을 누르면 거래가 시작됩니다. 장비 점검 중이거나 준비가 안 됐을 때는 눌러도 반응하지 않고, 화면 상단에 점검 안내가 표시됩니다.
>
> 🛠 `520×160px · 문구키 btn_start · 이미지 btn_start_2.png`

- **1행 이름** — 코드명 금지, 평이한 우리말 ("btnStart" ❌ → "시작하기 버튼" ✅)
- **2행 설명** — 기획 톤. "무엇이 일어나는가"를 사용자 관점으로. 이 서술은 **코드에서 확인한 실제 흐름**이다 — 추측이나 옛 기획서 복사가 아니라.
- **3행 🛠 스펙줄** — 크기·문구키·리소스명 등 개발 참고만 작은 글씨로 격리. 비개발 독자는 이 줄을 건너뛰어도 된다.

화면 고유 모달은 6항목 표준으로 기술한다:

| 항목 | 예시 (관리자 인증 모달) |
| --- | --- |
| 트리거 | 관리자 메뉴 진입 시 자동 표시 |
| 입력 | 관리자 카드 태그 또는 비밀번호 |
| 내용 | 안내 문구 + 가상 키패드 |
| 액션 | 확인 / 취소 |
| 정책 | 3회 실패 시 초기 화면 복귀 |
| 결과 | 성공 → 관리자 메뉴, 취소 → 이전 화면 |

문서 하단에는 **디스크립션 TBD 표** — 코드만으로 알 수 없는 정책(금액 한도의 근거, 문구의 의도 등)을 번호별 빈칸으로 남기면, 담당자가 채우고 확정 반영하는 협업 루프가 돈다.

## 파일럿 — 산출을 넘어 리뷰까지

역기획 산출이 쌓이자 다음 실전은 운영 모니터링 도구의 **기획 리뷰**였다. AI 기획자가 소스와 화면을 읽고 기획 관점의 갭 9개를 문서로 산출했다 — '이 버튼은 누르면 무슨 약속을 지키는가'를 개발자가 아닌 사용자 관점으로 짚는 문서다.

## 시행착오

- **역할 뭉갬** — 파일럿에서 기획자가 발견한 갭의 개발까지 같은 세션이 겸해버렸다. 이번만 마무리하고 다음부터 분리로 확정 — 역할 경계는 규칙 문서가 아니라 실전 사고에서 확정된다.
- **미러 복사 잔재 사고** — 대칭 제품 화면을 clone해 라벨만 치환했더니 도메인 고유 동작 서술이 원본 제품 것으로 남았다. '동작 서술은 항상 타깃 소스로 재검증'을 헌법에 반영.
- **"소스에 없음" 오판** — 템플릿 폴더만 보고 화면이 없다고 판단했는데, 실제로는 다른 계층(JSP)에 있었다. 부재 결론은 전체 구조를 본 다음에만.
- **빌드 주의 누적 → 규칙화** — 카드 프레임 기본 클리핑이 모서리 배지를 자르는 등 Figma 빌드의 자잘한 함정을 만날 때마다 누적 규칙으로 문서화 — 다음 화면부터는 안 밟는다.
- **말투도 설계 대상** — 기획자 산출물은 압축·기술 말투 금지, 쉬운 요약+비유+논리사슬 의무. 같은 AI라도 역할에 따라 커뮤니케이션 규격이 달라야 한다.

## 결과

제품 3종의 기획 문서 공백이 '지금 화면' 기준 스토리보드로 메워졌다. 비개발 구성원도 리뷰할 수 있는 톤이라 기획·QA·신규 입사자의 공통 참조점이 되고, 플랫폼화 시 baseline으로 재사용된다. 그리고 이 산출 규격 전체가 AI 기획자의 방법론 층에 계약으로 남아 — 다음 제품, 다음 세션에서도 같은 규격으로 반복할 수 있다.

> 기획자의 본질은 커뮤니케이션이다. 산출물은 비개발 구성원도 이해하는 쉬운 요약·비유·논리사슬로 쓰고, 코드 용어는 격리했다. AI가 가장 잘하는 '읽고 재구성'이 역기획과 정확히 맞아떨어졌다.
