> 운영 알람을 자동 분류하고 미팅용 리포트까지 자동 생성하는 도메인. 매 배포 주기마다 반복 수행.

## 무엇을 만들었나

운영 이슈 분석 프로세스 전체를 자동화. 한 달 치 알람 약 600건(첫 적용 회차 실측 615건)을 사람이 수동 분류하면 반나절이 걸리던 일이다. 알람 export 데이터를 받아 6색 분류 규격에 따라 분류하고, 미팅용 리포트를 자동 생성하여 현장 담당자 회의에 바로 투입한다.

- **규격:** 분류 기준 문서 약 370줄 (단일 출처)
- **분류 자동화:** 분류 스크립트 → 분류 결과 시트
- **리포트 자동화:** 리포트 스크립트 → 슬라이드 18장
- **폴더 자체 완결:** 규격/스크립트/데이터/리포트를 한 폴더로 관리 — 다른 사이트 이관 시 통째로 복제 가능

## 6색 분류 체계

처리 방향 기준으로 이슈를 6가지로 자동 분류. 같은 처리 방향이면 같은 컬러.

- 🔴 **S/W 수정 필요** — 버그 / 알림 정책 / S/W 우회
- 🟠 **운영 미숙** — 교육·SOP 개선
- 🟢 **현행 유지** — 정상 알림 / 수정 불가
- 🟣 **H/W 이슈** — 물리적 조치 불가피
- ⚫ **환경 이슈** — 외부 요인
- 🟡 **논의·로그 분석 필요** — 어디에도 안 걸린 항목이 자동으로 모이는 catch-all(기본 분류)

## 어떤 문제를 겪었나

### 1. 분류 체계의 반복적 리팩토링

초반엔 8색(이슈 유형별)으로 시작 → 4색(처리 방향별)으로 전면 재편 → 6색으로 수렴. 관리 기준이 명확해지는 데 20회 가까운 버전 갱신이 필요했다.

- **해결:** 규격 문서에 버전 이력을 남기고 컬러 간 이동 원칙을 명문화.

### 2. 매칭 규칙이 케이스마다 다름

어떤 계열은 알람 코드 하나로 식별되지만, 어떤 계열은 한 코드 안에 여러 처리 방향이 혼재되어 5개 컬럼을 조합해서 매칭해야 했다.

- **해결:** 스크립트는 OR + AND 복합 조건을 모두 지원. 코드 체계 표준화 자체는 조직 차원의 과제로 분리해 공유 (QA 단독 추진 불가 영역).

### 3. 전문 용어와 실무 사이의 간극

'SSOT(Single Source of Truth)', '입도 불균일' 같은 엔지니어링 용어는 실무에서 의미 전달이 어려웠다. '기준 문서', '세분화 정도가 다름' 같은 평이한 표현으로 즉시 교체.

### 4. 리포트 UI 디테일 — 메모 박스 잘림

메모 박스 높이가 작아 안내 문구가 테두리를 침범했고, 배경 도형과 텍스트박스가 별도 레이어라 클릭해도 바로 편집이 안 됐다. 박스+텍스트를 통합하고 중앙 정렬로 해결. **'클릭하면 바로 타이핑'이 안 되면 미팅에서 도구를 쓰지 않는다.**

## 핵심 교훈

> 분류 규격과 코드를 분리하라. 규격(md)이 바뀌면 코드가 따라간다. 그 반대가 되면 지식이 코드에 갇힌다.

- 분류 체계는 처리 방향 기준으로 통합하라. '이슈 유형별'로 나누면 컬러가 계속 늘어난다.
- catch-all 규칙이 있어야 분류 누락이 0이 된다. 신규 알람은 자동으로 논의 대상이 된다.
- 문서는 사용자 용어로 써라. 엔지니어링 용어는 번역이 필요하다.
