> 외부 파트너사가 기술지원 히스토리를 직접 적재하고, 내부 이슈 트래커의 기술지원 프로젝트와 100% 양방향으로 동기화되는 웹 서비스(FSM, Field Service Management)를 구축해 운영 오픈했다. 로드맵 P0~P7 전량 완료.

## 배경

현장 장비의 기술지원 기록과 내부 이슈 트래커가 따로 놀면 대응 이력이 흩어진다. 외부 파트너는 내부 트래커에 접근할 수 없고, 내부는 현장에서 무슨 조치가 있었는지 실시간으로 모른다. 파트너가 올린 기록이 자동으로 내부 이슈가 되고, 처리 상태가 다시 파트너 화면에 돌아오는 단일 흐름이 필요했다.

북극성 목표는 한 문장으로 고정했다: "무료 호스팅 웹앱으로 외부 파트너사가 기술지원 히스토리를 적재하고, 추후 통계까지 활용한다. 방법은 이슈 트래커와 100% 매칭 연동." 모든 세션은 이 문서부터 읽고 시작했다.

## 시스템 구성과 작업 프로토콜

- 스택: React/TS 프론트(무료 호스팅, main push = 자동 배포) + 서버리스 Edge Function + 관리형 DB·스토리지. 전부 무료 티어 — 인프라 비용 0.
- 작업 프로토콜 고정: 구현 → md 문서화 → 테스트 페이지 오픈 → 사용자 확인 → feature 브랜치 push.
- 단일 출처 문서 2개: 로드맵(북극성·P0~P7·미결정 목록)과 시간순 진행기록. AI 세션이 바뀌어도 이 둘만 읽으면 맥락이 복원된다.

## 로드맵 P0~P7

| Phase | 내용 | 핵심 결정 |
| --- | --- | --- |
| P0 환경 | 자체 계정 백엔드 신설 | dev 프로젝트 생략, 운영 직결 + 전환기 호환 모드 |
| P1 값 체계 정합 | 마스터 재시드 + Severity·Status를 트래커 스케일로 전환 | "이름 = 트래커 값" 원칙 → 매핑 테이블 자체가 불필요해짐 |
| P2 이슈 생성 | 커스텀필드 5종 + 본문 템플릿 | 실측해보니 필수필드 4종 → 앱에 없던 Type 필드 신설 |
| P3 상태 동기화 | 7상태 10전이 실측 → BFS 경로탐색 multi-hop 자동 전이 | 드리프트 시 조용한 오동작 대신 명확한 실패 |
| P3.5 역방향 | 트래커→앱 상태 역동기화 | Automation 월 한도 회피 → 실행 무제한 웹훅 채택 |
| P4 사진 첨부 | attachments API multipart 업로드 (best-effort) | URL 본문 삽입 → 정식 첨부로 전환 |
| P5 통계 | DB 집계 RPC + 재발장비/적체 대시보드 + 월간 리포트 | 차트 라이브러리 대신 SVG 직접 |
| P6 접근 제어 | 가입 승인제 + role + 사이트 격리 + private 버킷 | 외부 오픈 전 필수 보안 게이트 |
| P7 운영 전환 | repo·호스팅·백엔드 소유권 일원화 → 운영 오픈 | 타인 계정 의존 0 — 셀프서브 구조 확립 |

## 연동 설계 — 양방향을 지탱하는 원칙들

- **"이름 = 트래커 값" 원칙** — 앱의 마스터 데이터 이름을 트래커 옵션과 글자 단위로 일치. 매핑 테이블이 통째로 사라지고 파트너의 용어 학습 비용도 0.
- **이슈 생성** — 커스텀필드 5종 + 장비 ID 전송. 본문은 트래커 기본 템플릿과 동일한 색상 패널 구조를 직접 그려 넣음 — API 생성 이슈는 템플릿이 자동 적용되지 않기 때문. 사진은 생성 후 정식 첨부(실패해도 기록 저장은 유지).
- **상태 동기화** — 워크플로우를 탐사 이슈로 실측해 7상태 10전이 그래프 확보, 상태 버튼 클릭 시 BFS 최단 경로로 자동 전이. 각 hop마다 실제 전이 목록과 대조 — 워크플로우가 바뀌면 명확히 실패하도록 설계.
- **역방향** — Automation은 월 실행 한도가 사이트 공유라 탈락, 실행 무제한인 시스템 웹훅 채택. HMAC 서명 검증 + 멱등이라 루프 없음.
- **소유권 원칙** — 기록 본문(현상/조치/교체부품)은 앱 소유라 역동기화 안 함(원본 무결성), 진행 상태는 트래커 소유로 양방향.
- **과거 데이터 Import** — 실시간 대신 관리자 수동 pull(충돌 회피). 본문 섹션 헤더를 파싱해 필드로 복원, 멱등 설계.
- **마스터 Sync** — 초기엔 이름 매칭이라 rename 시 데이터가 고아가 되는 사고 → 옵션 id 매칭으로 전환해 rename 자동 추적. 삭제는 참조 0일 때만.
- **실작성자 매핑** — 토큰 계정이 전부 reporter가 되던 문제를 사용자↔계정 매핑으로 해결.

## 접근 제어와 보안 — 외부 오픈의 전제 조건

- 가입 승인제 — 가입은 열되 승인 전 데이터 접근 0. 역할 기반 권한.
- 사이트별 접근 격리 — 파트너가 담당 사이트 외 이슈를 못 보게 DB 레벨(RLS) 강제. UI 가드는 UX 레이어일 뿐, 강제는 서버층 — JWT 시뮬레이션으로 격리 완전성 검증.
- 사진 버킷 private 전환 — 만료형 signed URL, 구 공개 URL 차단 네거티브 검증까지.
- 자체 보안 감사 → 백로그 운영 — 웹훅 HMAC 승격, CORS 화이트리스트, 로그에서 고객 데이터 제거, 마지막 관리자 보호 트리거 등.

## 시행착오

- **막다른 상태 발견** — 워크플로우 실측 중 나가는 전이가 없는 상태를 발견, 워크플로우 자체를 수정했다. 연동을 만들다 보면 상대 시스템의 설계 결함이 먼저 드러난다.
- **rename 고아 데이터 사고** — 옵션 이름을 바꿨더니 기존 레코드가 구 이름에 박제되어 이슈 생성이 깨졌다. 데이터 이전으로 복구하고 id 매칭으로 재발 방지.
- **한글 인코딩 사고** — 셸 일괄 치환이 UTF-8 한글을 잘못 읽어 5개 파일이 깨짐. git 복구 후 전용 편집 도구로만 치환하는 수칙을 영구 등록.
- **무료 플랫폼 제약을 설계로 우회** — Automation 한도 → 웹훅, 배포 author 제한 → git identity 고정, 차트/PDF 라이브러리 → SVG 직접/브라우저 인쇄. 제약이 오히려 단순한 설계를 강제했다.
- **스코프 재정렬** — 1차 기술지원 플랫폼으로 컨셉을 좁히면서 Root Cause(1차에서 원인 단정 불가)·Downtime(기록 시점 측정 불가) 필드를 과감히 제거. 폼이 가벼워야 현장에서 쓴다.

## 결과

로드맵 P0~P7 전량 완료, 운영 도메인 오픈. 파트너가 기록을 저장하면 자동으로 이슈가 생기고(필드·본문·사진 포함), 내부에서 처리 상태를 바꾸면 수 초 내 파트너 화면에 반영된다. 장비별 재발 통계와 월간 리포트까지 — 기술지원 이력이 '흩어진 기록'에서 '집계 가능한 자산'이 됐다.

> 양방향 동기화의 핵심은 기술이 아니라 소유권 설계다. '이 데이터는 누가 주인인가'를 먼저 정하면 충돌·루프·덮어쓰기 문제가 설계 단계에서 사라진다.
