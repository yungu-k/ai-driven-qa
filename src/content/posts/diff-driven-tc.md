> 릴리스 TC를 이슈 티켓 산문이 아니라, 직전 검증 버전과 최신 코드의 git diff로 설계하는 방법론을 만들었다.

## 배경

릴리스 때마다 '이번에 무엇이 바뀌었나'를 이슈 티켓으로 파악했습니다. 그런데 티켓에 안 적힌 변경, 시그니처만 바뀐 회귀 포인트는 산문에 안 보입니다. 가장 정직한 변경 목록은 코드 자체입니다.

## 접근

- 기준 버전 ↔ 최신 코드 git diff → 동작 변경 추출 → 신규/변경/회귀 분류 → 릴리스 TC. AI가 diff를 읽고 영향 범위를 초안하면 사람이 검증합니다.
- **baseline 원칙:** 운영 배포 버전이 아니라 '마지막 QA 검증 버전'. 그래야 diff = 미검증 분만 남음.
- **TC 문구는 테스터 언어:** 함수명·필드명 금지, 메뉴·화면·조작·기대 결과로 풀어서 작성.

## 시행착오

- **baseline 오설정 사고** — 기준을 운영 배포로 잡았다가 이미 검증 완료된 기능을 신규로 오인, TC 7건 썼다가 전부 삭제. '검증완료 이력 먼저 대조'를 규칙으로 승격.
- **git의 한계** — 성격 다른 변경이 한 커밋에 번들되거나 버전 태그 부재 시 기계적 분리 불가 → 사람 판별로 경계 명시.

## 결과

실제 릴리스에 적용해보니 diff의 가치는 양방향이었다. 한 제품은 diff가 '다국어 문구만 변경, 로직 무변화'를 증명해 TC를 통째로 스킵 — **테스트를 안 하는 결정도 근거를 갖게 됐다**. 다른 제품은 신규 기능 TC로 이어졌다. 방법론은 TC 설계 가이드의 정식 장으로 편입되었고, [AI QA 팀원](/posts/ai-qa-engineer-teammate)의 3대 역할 중 하나가 됐다.

> '무엇을 테스트할지'의 근거가 사람의 기억·티켓 산문에서 검증 가능한 소스(코드)로 바뀌었다. 단, 코드에서 나온 TC라도 읽는 사람은 테스터 — 언어는 독자를 따른다.
