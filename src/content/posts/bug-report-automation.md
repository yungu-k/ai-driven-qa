> TC에서 Fail 항목을 선택하면 이슈 트래커에 버그를 자동 생성하고, TC 시트를 자동 업데이트하는 워크플로우.

## 무엇을 만들었나

Bug Report 작성 규칙을 규격 문서에 정의하고, TC 시트의 정보(사전조건·테스트 스텝·기대 결과)를 기반으로 이슈를 자동 생성하는 프로세스를 구축했다.

- Summary: `[라벨코드] 화면명 - 현상 요약` 형식 자동 적용
- Description: 환경/사전조건/재현절차/기대결과/실제결과/관련TC 6개 섹션 자동 구성
- Epic·라벨·테스트 버전을 TC 시트 상단에서 자동 파싱
- 버그 생성 후 TC 시트에 Fail + 이슈 URL 자동 입력
- 프로젝트별 라벨/Epic만 변경하면 어떤 프로젝트에든 적용 가능

## 어떤 문제를 겪었나

### 1. 라벨 코드 vs 풀네임 혼동

처음에는 Summary에 풀네임을 사용했지만, 실제 Epic의 라벨 코드를 그대로 써야 팀 내 검색/필터가 동작했다. 규칙을 명시하여 해결했다.

### 2. 테스트 버전 캐싱 문제

테스트 버전을 대화 시작 시 한 번만 읽어서 캐싱하면, 중간에 버전이 바뀌었을 때 이전 버전으로 버그가 생성되었다. '매번 실시간 조회' 규칙을 피드백 메모리에 저장하여 해결했다.

### 3. environment 필드 제한

이슈 트래커의 environment 필드는 특수 문서 포맷을 요구해 직접 사용이 불가능했다. Description 본문 내 환경 섹션으로 대체하는 방식으로 해결했다.

## 핵심 교훈

> 자동화의 가치는 '버그 하나 만드는 시간'이 아니라, '50개 버그를 일관된 포맷으로 만드는 시간'에 있다.

- 라벨 상속 규칙(Epic에서 라벨 자동 상속)처럼, 팀 규칙을 코드화하면 휴먼 에러가 사라진다.
- TC ↔ 이슈 트래커 양방향 연동이 핵심. 버그 생성 시 TC 업데이트, 이슈 상태 변경 시 TC 동기화.
