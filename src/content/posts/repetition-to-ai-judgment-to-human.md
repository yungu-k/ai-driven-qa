> QA 자동화 여정을 외부 독자 관점에서 정리한 글입니다. 각 Phase의 배경·접근·시행착오·결과를 구체적으로 풀어서 서술합니다.

## 목차

## 왜 QA가 자동화에 뛰어들었나

QA 엔지니어는 종종 '반복 작업이 많지만 판단이 중요한' 애매한 자리에 서 있습니다. Test Case 작성, Bug Report 기록, Sign-off 리포트 구성 — 이 반복을 AI에게 맡기고, 제가 판단과 전략에 집중할 수 있도록 경계를 다시 그어보기로 했습니다. Claude Code를 실무에 밀어 넣어 QA 업무 전체를 재구성한 기록입니다.

결론부터 말씀드리면 **결과보다 구조가 중요했습니다**. 구체적으로는 여러 개발 도구를 MCP(Model Context Protocol)로 엮어 단일 AI Agent가 전체 흐름을 다루게 한 것, 그리고 노하우를 코드가 아니라 규격 문서(md)에 담은 것.

> 경험은 기록해야 내 것이 된다. 그리고 공유해야 누군가에게 도움이 된다.

## 여정을 관통한 구조 — MCP로 엮은 작업 흐름

본격적인 Phase별 이야기로 들어가기 전에, 여정을 관통한 공통 구조를 먼저 말씀드립니다.

### 기존 방식의 한계 — 맥락의 단절

자동화가 없던 시절의 흐름은 이랬습니다. Figma에서 시안을 보고 → Google Sheets를 열어 TC를 쓰고 → Jira로 넘어가 버그를 등록하고. 매 도구 전환마다 컨텍스트가 끊깁니다. '내가 지금 뭘 하고 있었지?'를 머릿속에서 다시 구성해야 합니다.

### MCP 연결 — 하나의 Agent에서 전부 제어

아래 5개 시스템을 단일 AI Agent에서 제어하도록 연결했습니다.

| 시스템 | 연동 방식 | 역할 |
| --- | --- | --- |
| Jira | MCP | 버그 조회·생성·상태 전이, Epic·라벨 관리 |
| Confluence | MCP | 회의록 작성, 프로세스 가이드 공유 |
| Google Sheets | Python API | TC 시트·Sign-off·Status 리포트 작성 |
| Figma | REST API + MCP | 시안 메타데이터·스크린샷·노드 diff |
| Notion | MCP + REST | 회고 기록, 이 블로그의 원본 |

### MCP를 쓰면서 배운 점

- **완벽한 자동화가 아니라 맥락 유지가 중요하다** — 작업을 사람이 누르든 AI가 누르든, '같은 맥락에서 이어진다'는 감각이 생산성을 결정합니다.
- **각 MCP의 한계는 여전히 존재한다** — 일부 필드는 특수 포맷을 요구하고, 일부 서식은 API로 제어 불가능합니다.
- **단일 기준 문서가 있을 때 시너지가 크다** — 여러 시스템에 나뉘어 쓰이는 규칙도 기준 문서(md) 한 곳에 있어야, AI가 시스템을 넘나들며 일관되게 움직입니다.

## Phase 1 — TC 설계 자동화: 디자인 시안에서 코드 diff까지

### 배경

신규 화면이 나올 때마다 디자인 시안을 보며 Google Sheets에 TC를 한 줄씩 써 내려갑니다. 화면 하나에 30\~60분, 프로젝트 하나에 수십\~수백 화면.

### 1막 · 디자인 시안을 읽다

'프롬프트로 잘 시키기'보다 '**규격 문서를 먼저 만들기**'로 방향을 잡았습니다. TC의 필수 필드, 서식 규칙, 작성 톤을 규격 문서에 정의하고 AI가 그 규격대로 산출하게 했습니다.

디자인 파일에서 정보를 뽑을 때는 '메타데이터 우선 + 핵심 화면만 스크린샷 보완' 하이브리드 방식을 썼습니다. 서식 체크리스트 9단계, API 한계와 수동 영역의 경계 긋기 같은 세부 시행착오는 [TC 설계 자동화](/posts/tc-design-automation/)에 담았습니다.

### 2막 · 코드 diff를 읽다 — 영향도 기반 릴리스 TC

신규 화면 TC는 디자인 시안이 출발점이지만, 릴리스 TC는 '이번에 무엇이 바뀌었나'가 출발점입니다. 이슈 티켓의 산문만 믿으면 티켓에 안 적힌 변경·회귀 포인트를 놓칩니다. 가장 정직한 변경 목록은 코드 자체 — 그래서 직전 검증 버전과 최신 코드의 git diff로 영향 범위를 산정해 TC를 설계하는 방식을 더했습니다.

가장 아픈 사고는 **baseline 오설정**이었습니다. 기준을 '운영 배포 버전'으로 잡았더니 이미 지난 차수에서 검증 완료된 기능이 통째로 '신규'로 잡혀, TC 7건을 썼다가 전부 삭제했습니다. 교훈: baseline은 운영 배포가 아니라 **'마지막 QA 검증 버전'** — 그래야 diff = 미검증 분만 남습니다. diff 한 줄이 TC 세 개가 되는 실제 변환 과정은 [코드 diff 기반 영향도 TC 설계](/posts/diff-driven-tc/)에.

### 결과

신규 화면 1개당 TC 작성 시간이 크게 단축됐고, 동일 규격을 따르므로 리뷰어도 핵심에만 집중할 수 있게 됐습니다. 그리고 TC 설계의 입력이 하나 늘었습니다 — 신규 화면은 디자인 시안을 읽고, 릴리스는 코드 diff를 읽습니다. '무엇을 테스트할지'의 근거가 사람의 기억·티켓 산문에서 검증 가능한 소스로 바뀌었습니다.

## Phase 2 — Bug Report 자동화

### 배경

TC를 수행하다 Fail이 나오면 Jira에 버그를 등록하고, 다시 TC 시트로 돌아와 결과를 Fail로 바꾸고 이슈 링크를 붙여 넣는 왕복 작업.

### 접근

TC 시트에 이미 재현 절차·기대 결과·사전 조건이 있으니, 이를 버그 Description의 기본 재료로 활용했습니다. 버그 생성이 끝나면 TC 시트로 돌아와 결과를 Fail로, 생성된 이슈 URL을 자동 입력합니다.

### 결과

50건의 버그를 일관된 포맷으로 등록하는 게 핵심이었습니다. 사람 손으로는 뒤로 갈수록 포맷이 흐트러지는데, 자동화는 마지막 1건까지 같은 품질입니다. 라벨 규칙과 버전 캐싱에서 밟은 삽질의 전말은 [Bug Report 자동화](/posts/bug-report-automation/)에 있습니다.

## Phase 3 — QA Report 자동화 (Sign-off + Status)

### 배경

QA 종료 시점의 Sign-off 리포트와 진행 중의 상태 리포트는 숫자·표·조건부 서식이 복잡한 문서입니다.

### 접근

두 리포트의 레이아웃을 규격 문서로 뽑아냈습니다. Test Summary 자동 집계, Defect Summary 수집, Verification Details 구성까지.

### 결과

이 Phase에서 가장 많은 자잘한 사고를 겪었고, 그 끝에 배운 것은 — 가장 어려운 것은 '데이터 수집'이 아니라 '**서식 일관성**'이라는 사실입니다. 머지 순서, 행 참조, 빈 행 하나가 전체 레이아웃을 무너뜨립니다. 사고들의 목록과 해결은 [QA Report 자동화](/posts/qa-report-automation/)에.

## Phase 4 — 디자인 변경 감지 + TC 커버리지 갭 분석

### 배경

프로젝트가 진행되면 디자인 시안은 계속 업데이트됩니다. 범위가 커지면 어느 화면이 바뀌었는지 추적이 어렵습니다.

### 접근

Figma REST API로 페이지별 노드 트리를 JSON 스냅샷으로 저장하고, 이전 스냅샷과 diff로 '추가/삭제/변경된 프레임'을 자동 식별합니다.

### 결과

'timestamp 비교'보다 '실질적 diff'가 정확하다는 일반화된 패턴을 얻었습니다. 무엇이 오탐을 만들었고 어떻게 걸러냈는지는 [디자인 변경 감지](/posts/design-change-detection/)에서 다룹니다.

## Phase 5 — Regression + Jira 상태 동기화

### 배경

Regression 회차 때마다 Full TC에서 P0를 찾아 복사하는 작업이 필요했고, Jira의 상태 변화가 TC 시트에 수동으로만 반영됐습니다.

### 접근

첫 번째는 Regression 추출 — 단순한 'P0만 추출'이 아니라 '플로우 완전성을 위해 선행 단계의 P1도 선별 추가'라는 규칙. 두 번째는 상태 동기화 — '완료 → Pass', '반려 → N/A', '수정됨 → Fail 유지(검증 대기)' 규칙을 정의해 자동 반영.

### 결과

상태 동기화에서 가장 중요한 건 '**상태 전이 규칙을 명확히 정의하는 것**'입니다. 모호한 규칙은 데이터 불일치로 이어집니다. 다중 이슈 연결 같은 경계 사고는 [Regression TC 추출 + 상태 동기화](/posts/regression-and-status-sync/)에.

## Phase 6 — 운영 모니터링: 수동 분류에서 실시간 대시보드까지

### 배경

Phase 5까지가 TC·리포트라는 QA 내부 산출물의 자동화였다면, 여기서부터 무대가 바뀝니다 — 개발 단계가 아니라 이미 배포되어 돌아가는 현장 운영입니다.

현장에서 발생하는 이슈 알람은 외부 채널로 실시간 수집됩니다. 배포 주기마다 '어떤 이슈가 나왔고, 어떤 건 우리가 수정할 일인지'를 판단해 미팅 자료로 만들어야 했습니다. 한 달 치 알람이 약 600건 — 수동 분류에 반나절. 더 큰 문제는 기준이 사람 머리에만 있었다는 것.

### 1막 · 분류 체계를 세우다

'어떤 부서가 처리할 일인가' 기준으로 처리 방향별 6개 컬러를 정의했습니다.

- 🔴 S/W 수정 필요 — 개발팀이 조치할 영역
- 🟠 운영 미숙 — 교육·SOP 개선으로 해결할 영역
- 🟢 현행 유지 — 정상 알림이거나 조치 불가 영역
- 🟣 H/W 이슈 — 물리적 조치가 불가피한 장애
- ⚫ 환경 이슈 — 외부 전기·네트워크 등
- 🟡 논의·로그 분석 필요 — **어디에도 안 걸린 항목이 자동으로 모이는 catch-all(기본 분류)**

이 catch-all 규칙이 핵심입니다. 규칙에 없는 새 이슈가 '미분류로 증발'하지 않고 자동으로 논의 대상이 됩니다. '분류 누락 0건'이 구조적으로 보장됩니다.

이 체계가 한 번에 나온 건 아닙니다 — 분류 색이 수렴하기까지의 리팩토링, 용어 평이화, 알람 식별 체계처럼 QA 단독으로 풀 수 없던 문제는 [운영 이슈 트리아지 자동화](/posts/live-issue-triage/)에 담았습니다.

### 2막 · 실시간 대시보드로

분류 리포트는 강력했지만 사후적이고 정적이었습니다 — 배포 주기마다 한 번, 미팅 자리에서만 펼쳐 봤고, 정작 장비가 멈추는 그 순간은 아무도 실시간으로 보지 못했습니다.

그래서 같은 알람 데이터를 실시간으로 읽는 전용 웹 대시보드를 구축했습니다.

- **상태 상황판(wallboard)** — 장비별 최신 이벤트로 색을 판정(정상 초록·연결끊김 회색·심각 빨강·잔량경고 노랑). 관제실 대형 모니터에 상시 띄우는 패널, 자동 새로고침.
- **분석 뷰** — 기간·장비·상태로 걸러 알람 추이를 차트와 표로.
- **가용성(Downtime) 뷰** — 장비별 가동률 중심, 막대를 누르면 그 구간의 장애가 분해. 보고용 PDF 내보내기.

핵심은 분류 기준을 코드 한 곳에 모은 것 — 새 알람 유형이 생기면 그 한 파일만 고치면 네 화면(상태·분석·이슈·가용성)에 동시 반영됩니다. 대시보드 구축 과정은 [실시간 키오스크 모니터링 대시보드](/posts/realtime-kiosk-monitoring/)에.

### 결과

배경에서 '약 600건'이라 어림잡았던 그 한 달 치 알람 — 실측 615건 — 을 누락 0건으로 자동 분류하고 리포트 18장을 한 번의 명령으로 생성하는 파이프라인, 그리고 이것이 실시간 모니터링으로 진화했습니다. '배포 주기마다 한 번 보는 정적 리포트'에서 '언제든 열면 지금 상태가 보이는 살아있는 모니터'로 — 같은 데이터라도 사후 정리에서 상시 관제로 쓰임이 바뀌었습니다.

## Phase 7 — AI를 동료로: 반복에서 팀으로

### 배경

Phase 1~6은 '내 반복 작업을 AI에 위임'하는 여정이었습니다. 여기까지 오니 다음 질문이 생겼습니다 — 반복을 넘어 '판단이 필요한 일'까지 AI가 동료로 거들 수 있을까? QA를 넘어 기획·운영 도메인으로도 같은 방식이 통할까?

### 접근

두 방향으로 확장했습니다. 하나는 **'AI를 팀원으로'** — 기획자(무엇을 만들까)·QA 엔지니어(무엇이 깨질까)·개발자(어떻게 구현하고 배포하나) 세 팀원을 만들어 핸드오프로 연결하고 서로 검수하게 했습니다. 자세한 설계는 [AI 팀원 만들기 3부작](/series/ai-teammate/)에.

다른 하나는 **'자동화를 인접 도메인으로'** — 소스에서 화면 기획서를 복원하는 역기획 스토리보드, [기술지원 이력과 Jira의 양방향 연동](/posts/field-service-jira-sync/), [명령어 없이 버튼으로 돌리는 API 실행 Web UI](/posts/api-runner-web-ui/)까지, QA 밖의 문제에 같은 방식을 적용했습니다.

### 결과

'반복은 AI에게, 판단은 사람에게'의 경계가 사람 사이 협업 구조로 확장됐습니다. AI가 초안·검증·복원을 맡고 사람이 결정·승인·전략을 쥐는 분업이 QA를 넘어 기획·운영까지 퍼졌습니다. 공통 원리는 그대로였습니다 — 규칙은 코드가 아니라 문서에 먼저, 불변 원칙과 가변 상태를 분리, 독자에 맞춰 언어를 바꾼다.

## 지향하는 QA 모습

TC 서식 적용, 버그 포맷 통일, 리포트 수치 집계, 알람 분류처럼 '정답이 있는 반복'은 AI가 맡고, 테스트 전략·엣지 케이스 발굴·릴리스 Go/No-Go처럼 '상황에 따라 달라지는 판단'은 사람이 쥔다 — 이 경계 긋기가 여정의 출발점이자 결론이었습니다. 그 위에서 네 가지 원칙이 남았습니다.

### 노하우를 코드에 가두지 않는다

경험이 if-else 한 줄로만 남으면, 1년 후 같은 코드를 읽는 사람은 왜 그렇게 짰는지 알 수 없습니다. 그래서 모든 Phase에서 규칙은 코드가 아니라 md 문서에 먼저 썼습니다. 문서가 지식의 1차 저장소이고, 코드는 그 실행체입니다.

### 실패는 규칙으로 옮긴다

이 여정의 거의 모든 규칙은 한 번 이상 같은 실수를 반복한 끝에 나왔습니다. 같은 실수를 두 번 하지 않으려면 **개인의 기억이 아니라 문서**가 필요합니다.

### QA가 할 일과 조직이 할 일을 구분한다

내 자동화를 복잡하게 만드는 근본 원인이 조직 차원의 문제일 때 — 예컨대 알람 코드 표준화처럼 여러 부서가 합의해야 하는 일 — QA가 혼자 밀어붙일 수는 없습니다. QA는 그 문제를 데이터로 드러내 공유하고, 결정과 추진은 테크리드에게 넘깁니다.

### 도구는 채택되어야 의미가 있다

아무리 잘 만든 자동화도 동료가 쓰지 않으면 의미가 없습니다. 작은 UX 디테일 — 클릭 즉시 편집되는지, 서식이 깨지지 않는지 — 에 신경 썼습니다.

> 자동화는 결국 '사람이 판단에 집중할 수 있게 해주는 도구'이지, 사람을 대체하는 게 아닙니다.

## 다음 여정

자동화 파이프라인을 넘어 AI를 동료로 들이는 Phase 7까지 왔지만, 여전히 '채택'이라는 숙제가 남습니다. 도구를 만드는 것과 동료들이 그 도구를 실제로 쓰는 것은 다른 문제입니다. 다음은 팀 확산입니다 — 지금까지의 규격 문서·스크립트·AI 팀원이 다른 QA 엔지니어에게도 효과를 내도록 온보딩 가이드와 실습 템플릿을 준비하는 단계입니다.

그리고 이 여정 자체를 이렇게 외부에 공유합니다. 내부 기록(회고)과 외부 공유(블로그)는 서로 다른 형태가 필요하다는 것을, 이 글을 쓰면서 다시 배우고 있습니다.
