> 키오스크 앱(현금↔칩 교환 무인기)을 Web UI와 API 두 트랙으로 자동 검증하는 인프라. 소스를 받아 테스트를 생성·실행하고, 실패를 리포트·티켓으로 연결하는 과정을 한 번의 명령으로 반복한다.

## 무엇을 만들었나

- **Web UI 트랙 (Playwright):** 화면 진입·사용자 흐름·다국어·시각 회귀 검증. 키오스크 2기종 약 32개 화면.
- **API 트랙 (vitest + supertest):** 디바이스 서비스 모듈 18종 + 백엔드 API의 응답 계약(contract)·에러 코드 정합 검증.
- **Orchestrator (Node + TypeScript):** 소스 clone → 테스트 생성 → 원격 실행 → 리포트 → 티켓 등록 5단계 파이프라인.

| 트랙 | 검증 목적 | 도구 |
| --- | --- | --- |
| Web UI | 화면 노출·사용자 흐름·다국어·시각 회귀 | Playwright |
| API | 모듈별 응답 계약·에러 코드 정합 | vitest + supertest + zod |

## 자동화가 실제로 잡아낸 결함

- **응답 타입 불일치:** 일부 endpoint가 숫자여야 할 상태값을 문자열로 반환 → 프런트 비교 로직 우회 가능. 전수 number로 통일.
- **입력 검증 부재:** 음수·누락·null 파라미터가 모두 200으로 통과 → 검증 도입으로 400 정상화.
- **Content-Type 누락:** 한 endpoint만 text/plain 반환 → 거래 흐름 파싱 실패 원인. 수정 확인.
- **접근성 위반:** 버튼 라벨·이미지 대체텍스트 누락 등 자동 검출.

## 겪은 문제와 해결

### 1. Mock과 실제 응답의 부정합

디바이스 없는 환경에서 Mock 응답으로 테스트하다 보니 Mock과 실제 하드웨어 응답이 미묘하게 달랐다.

- **해결:** 에러 코드 카탈로그를 단일 출처로 정리하고, Mock에 시나리오 트리거를 심어 정상/에러 경로를 모두 재현. 발견한 부정합은 검출에 그치지 않고 코드 수정까지 연결.

### 2. 상시 빨간 fail → green baseline

초기엔 fail이 20건 넘게 쌓였는데, 점검 결과 단 하나도 실제 운영 결함이 아니었다(Mock 한계·타이밍). 상시 빨간 fail은 회귀 신호를 가리는 noise다.

- **해결:** 환경 한계 케이스를 manual-required로 분리하고, 안정적으로 통과하는 spec만 green baseline으로 묶어 회귀 게이트 신뢰도 확보.

### 3. 환경 종속성 → Local-first 전환

실 키오스크에 의존하면 장비 점유·전원 상태에 자동화가 묶였다.

- **해결:** 앱을 로컬에서 직접 기동하도록 전환. 실 장비는 최종 sanity check에만 사용.

### 4. 비기술 사용자 접근성

CLI는 운영·비개발 인원이 쓰기 어려웠다.

- **해결:** 브라우저에서 모듈 그룹을 골라 실행하고 실시간 로그·결과를 보는 Runner Web UI 추가.

## 핵심 교훈

> 자동화의 가치는 '통과'가 아니라 '신뢰할 수 있는 실패'에 있다. 항상 빨간 화면은 회귀 신호를 죽인다.

- 우회(Mock 변조)는 Web UI 검증엔 허용하되, API 정합 검증은 진짜 응답으로. 트랙별 책임을 분리해야 결과를 믿을 수 있다.
- 에러 코드·스펙은 단일 출처 문서로. 코드가 문서를 따라가야 지식이 코드에 갇히지 않는다.
- 부정합은 검출에서 멈추지 말고 수정까지. 자동화가 실제 코드 품질을 끌어올린다.

## 산출물 규모

- API 자동화: 18개 디바이스 모듈 + 백엔드 → 약 1,160 테스트 / 1,133 PASS / 0 fail
- Web UI 자동화: 2기종 32개 화면 × 4종 검증 = 약 128 spec
- 자동 검출·수정한 production 부정합: 5종
