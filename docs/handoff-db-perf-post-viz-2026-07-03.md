## 사전 판단 (기획자)

**어투/톤은 바꾸지 않는다.** 이 글은 평서체 일관, 과장 없음(오히려 "증명하지 않은 것" 섹션은 블로그 정체성의 모범), 용어 병기(시딩·논리적 읽기·풀스캔·커넥션 풀)·도구 실명(MS SQL Server 2022)까지 이미 갖췄다. 억지 수정 금지.

**핵심 = 시각화.** 결과 표 2개가 극적인 숫자를 담았는데 표라서 안 와닿는다. 여기에 그래프를 더한다.

## 📋 개발 핸드오프

### V1. 결과 1 그래프 — "무인덱스는 모든 조회가 7.5초"

- **무엇을**: "결과 1" 표(§ A/B/C) 위(또는 아래)에 **응답 시간 가로 막대그래프**를 더한다. 표는 그대로 둔다.
- **차트 사양 (기획 결정 — 이유 포함)**:
  - **측정값 = 응답 시간**(ms). 논리적 읽기가 아니라 시간을 그린다 — 초 단위는 비개발자도 즉시 체감. (논리적 읽기는 표에 남는다. **한 그래프에 두 축 금지**.)
  - **선형 축**(로그 아님). A(7,487ms)가 압도적으로 크고 B(1ms)·C(1ms 미만)는 **거의 0에 가까운 슬리버**로 보이는 게 정직하다 — 실제로 B·C는 무시 가능하게 빠르다는 게 메시지다. 로그축은 이 "차이"를 뭉개므로 쓰지 않는다.
  - **직접 라벨**: 각 막대 끝에 실제 값(7,487ms / 1ms / <1ms). 슬리버라도 숫자로 읽힌다.
  - 색: 재난 케이스(A 무인덱스)는 위험을 뜻하는 강조색, 정상(B·C)은 차분한 중립/양호색. **상태 인코딩이므로 라벨과 함께**(색만으로 구분 금지).
- **수용 기준**:
  - [ ] 결과 1 자리에 응답 시간 막대그래프가 있고, 표도 함께 있다.
  - [ ] A가 압도적으로 크고 B·C는 직접 라벨로 값이 읽힌다.
  - [ ] 다크/라이트 모두 정상, 색이 dataviz 검증(아래) 통과.

### V2. 결과 2 그래프 — "같은 인덱스, 대상만 바꾸면 절벽"

- **무엇을**: "결과 2" 표(§ B활성 / B극단 / C극단) 위에 **막대그래프**를 더한다. 이 글에서 가장 극적인 지점("10 → 1,178만")이라 시각화 효과가 가장 크다.
- **차트 사양**:
  - **측정값 = 응답 시간**(ms): B활성 1ms · B극단 107,176ms · C극단 1ms 미만. (V1과 같은 측정값으로 통일 — 독자가 두 그래프를 같은 잣대로 읽음.)
  - **선형 축**. B극단(107초)이 거대하게 솟고, B활성·C극단은 near-zero 슬리버 + 직접 라벨. "하나만 폭발하고 나머지는 한 자릿수 ms" 라는 구조가 시각으로 그대로 전달된다.
  - 색: B극단 = 위험 강조색, 나머지 = 중립. **특히 B활성과 B극단이 '같은 인덱스'임이 드러나게** — 같은 계열(B)인데 대상만 다르다는 게 요점. 라벨로 "B 단일(활성)" / "B 단일(극단)" / "C 복합(극단)" 명시.
- **수용 기준**:
  - [ ] 결과 2 자리에 막대그래프 + 표 공존.
  - [ ] B극단이 압도적이고, B활성/C극단은 라벨로 값이 읽힌다.
  - [ ] "같은 B 인덱스인데 대상만 다르다"가 라벨/배치로 드러난다.

### V3. (선택) 헤드라인 비교 콜아웃

- **무엇을**: 도입부나 결과 요약에 "before → after" 한 줄 콜아웃 1~2개. 예: **7,487ms → 1ms 미만 (약 7,500배)**, **논리적 읽기 189,016 → 4**. 큰 숫자 두 개 + 화살표 + 배율.
- **수용 기준**: [ ] 넣는다면, 근거는 본문 표의 실제 값과 일치. 없는 수치 지어내지 않음.
- 우선순위 낮음 — V1/V2 먼저.

### 인과 분리 표(§ "쿼리 성능 ≠ 시스템 장애")는 그래프로 만들지 않는다

정성 비교(거동/시스템 결과)라 표가 맞다. 그대로 둔다.

### 텍스트 미세 폴리시 (선택, 있으면)

- `worst-case` 첫 등장에 우리말 병기 — "worst-case(최악 조건)".
- `HikariPool`(§ 범위와 한계) — 앞서 "커넥션 풀"로 풀었으니, 여기선 "커넥션 풀(HikariPool)" 정도로 연결. 과하면 생략 가능.

## 구현 제약 (중요)

- **정적 블로그**다(AstroPaper, 런타임 차트 라이브러리 없음). 차트는 **인라인 SVG 또는 CSS 막대**로 — 새 JS 차트 라이브러리 도입 금지. 런타임 Node 0 유지.
- **dataviz 스킬을 따라 만든다**: form은 위에서 정함(가로 막대·선형·직접 라벨). 색은 **블로그 기존 토큰**(`--accent`, `--muted`, `--foreground`)을 기반으로, 재난 케이스용 위험색 1개를 추가하되 **`scripts/validate_palette.js` 로 light/dark 각각 검증**(눈으로 판단 금지). 마크 사양(둥근 4px 끝, 2px 간격, recessive 축, 텍스트는 색 아닌 잉크 토큰) 준수.
- **표는 유지**. 차트는 직관, 표는 정확한 값 + 접근성(table view). 둘 다 있어야 함.
- 본문 서술·구조는 (미세 폴리시 외) 건드리지 않는다.

## ✅ 구현 결과 (개발 세션, 2026-07-03)

**V1·V2** — 각 결과 표 **아래**에 CSS 가로 막대그래프(figure, 런타임 JS 0). 사양 준수: 응답 시간 단일 측정값·선형 축·전건 직접 라벨(값은 우측 고정 열, tabular-nums)·막대 18px(≤24px)·데이터 끝만 4px 라운드·텍스트는 잉크 토큰(시리즈색 텍스트 없음). 슬리버(1ms류)는 min 2px로 존재만 표시 — "하나만 폭발" 구조가 그대로 보임. V2는 B 두 행 연속 배치 + 캡션 "위 두 막대는 같은 B(단일) 인덱스, 대상만 다름". 상태색은 캡션("빨강 = 장애 상태/절벽 구간") + 행 라벨 병행 — 색 단독 인코딩 아님. 접근성: figure role="img" + aria-label 전체 수치 서술, 표 공존(table view), hover는 행 title 속성.

**V3** — 리드 인용 아래 콜아웃 2타일: "189,016 → 4" / "7,487ms → 1ms 미만". 본문 표 값과 동일(배율 표현은 지어내기 경계라 생략).

**텍스트 폴리시** — "worst-case(최악 조건)" 병기, "커넥션 풀(구현체는 HikariPool)" 연결.
