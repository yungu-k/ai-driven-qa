**4. 하이라이트는 함정 모음.** 독자가 실제로 얻어가는 부분이라 여기에 지면을 준다.

### 담을 내용 (섹션별)

**리드 인용(>)** — 한 문장으로 핵심 명제. 취지: *고객 요구와 무관하게 내부적으로 안정된 열차를 굴리고, 고객에게는 검증된 버전을 골라 배포한다.*

**1. 배경 — 무엇이 문제였나**
- 비정기 배포: 백로그를 모아뒀다가 고객이 원하는 시점에 일괄 배포.
- 그 결과 네 가지가 동시에 터졌다: 검증 끝난 변경분이 배포 못 되고 **쌓임** / 개발·QA 일정이 **고객 요청에 휘둘림** / 한 번에 다량 배포라 **회귀 범위·원인 격리 악화** / 오래 묵은 검증분의 **재검증 부담**.

**2. 방향 — 무엇을 분리했나 (글의 핵심 명제)**
- **내부 트랙**: 2~3주마다 항상 "배포 가능한" 검증 완료 버전을 만들어 둔다.
- **고객 트랙**: 고객이 원할 때 이미 검증된 버전 중 하나를 꺼내 배포한다.
- 즉 개발·QA를 **고객 일정의 예측 불가능성에서 분리**한 것. → **열차 비유**를 여기서 세운다(정시 출발, 못 탄 짐은 다음 열차, 승객은 원하는 열차를 골라 탐).

**3. 설계 — "왜"가 있는 결정만**
- **스프린트 길이 고정**(매번 고르지 않음). *왜*: 일정한 리듬이 있어야 속도 측정·캘린더 예측·QA 기간 산정이 가능.
- **작업일 배분표** — 이건 표로 넣는다(구체성이 값어치):

  | 스프린트 | 총 작업일 | 개발 | QA | └ 신규/변경 | └ 회귀 |
  | --- | --- | --- | --- | --- | --- |
  | 2주 | 10일 | 6일 | 4일 | 3일 | 1일 |
  | 3주 | 15일 | 9일 | 6일 | 4일 | 2일 |

- **Code Freeze = QA 착수일**(개발 종료일). 이후 신규 기능 머지 중단, 버그 수정만.
- **QA 순서: 신규/변경 먼저 → 회귀는 마지막.** *왜*: 마지막 회귀로 통합 영향도를 확인하기 위해.
- **범위 잠금(Scope Lock)**: 시작 후 추가 금지. 고객 긴급 건은 다음 스프린트나 급행(Hotfix) 트랙으로.
- **품질 게이트 + 정시 출발**: 게이트를 못 통과한 변경분은 그 스프린트에서 **빼고**, 열차는 **정시 출발**. 그리고 **고객 배포 승인 주체는 QA**("QA 승인 없이는 배포 불가"). ← 이 두 개가 이 설계의 성격을 규정하므로 강조.

**4. 버전을 둘로 나눈 이유 (비유 필수)**
- 문제: 모듈마다 버전이 제각각이라 릴리스 버전을 하나로 못 정함.
- 해법 2계층: **열차 번호 = 상자 겉의 이름표**(배포 단위) / **매니페스트 = 상자 안 물건 목록**(그때 어느 모듈이 어느 버전인지).
- 열차 번호는 의미를 가진 버전이 아니라 **묶음을 가리키는 포인터**. 모듈은 자기 버전 그대로, 바뀐 것만 올라간다.

**5. 반직관 지점 — 고객 배포용 버전을 따로 안 만든다**
- 소프트웨어는 **누적**이라 14번 열차 안에는 11~13번 변경분이 이미 들어 있다.
- 그래서 "11~14를 배포"가 아니라 **"14를 배포하면 앞의 것이 자동 포함"**.
- 고객 배포 = 새 버전 만들기가 아니라 **"어느 열차를 어느 고객에 적용했는지 기록"**. 고객마다 다른 버전이 라이브일 수 있다.

**6. 선반(Release Pool)이 실제로 되려면 — 전시회 시나리오**
- 상황: 개발은 15번 열차까지 갔는데, 전시회에선 11번 상태만 보여주고 싶다.
- 누적이라 **최신 코드로는 과거 상태를 못 만든다** → 시점을 고정해둬야 한다.
- 필요한 3종: **소스 시점 고정(태그)** · **그때의 버전 묶음 목록** · **검증된 설치본 보관**(다시 빌드하면 그건 검증 안 된 물건이다).
- 이 3개가 있어야 "선반에서 꺼내 쓰기"가 실제로 된다.

**7. 급행 트랙(Hotfix)** — 짧게 한 문단. 운영 심각 장애는 주기를 못 기다린다 → 라이브 기준 분기 → 최소 변경 → 범위 한정 회귀 → 긴급 배포, 그리고 **반드시 본류로 되돌려 병합**(안 하면 다음 열차에서 사라진다).

**8. 도구에 앉히기 (Jira) — 가장 큰 오해 교정**
- 전환 전: "릴리스 묶음"과 "QA"를 각각 **에픽**으로 만들어 관리하고 있었다.
- 바로잡은 계층(표로):

  | 개념 | 잘못 쓰던 방식 | 바로잡음 |
  | --- | --- | --- |
  | 반복 주기 | (없음) | 스프린트 = 2~3주 실행 단위 |
  | 기능 묶음 | 릴리스 버킷으로 오용 | 에픽 = 여러 스프린트에 걸치는 기능 테마 |
  | 릴리스 버전 | 에픽으로 관리 | 버전 필드 = 배포 단위 |
  | QA | 릴리스마다 QA 전용 에픽 | 에픽 폐지 → 각 작업의 흐름 안으로 |

- 명제 2개를 문장으로 살릴 것: **"에픽은 스프린트 담는 그릇이 아니다"**(에픽:스프린트는 N:M — 1:1로 묶으면 에픽이 '이번에 할 일 폴더'로 전락) · **"QA는 별도 단계가 아니라 각 작업의 완료 조건"**(옆 라인으로 빼두면 워터폴로 되돌아간다).
- **버그는 독립 티켓**으로 두고 원인 작업에 연결. *왜*: 자체 우선순위·수정이 실릴 버전·재오픈 관리가 필요한데, 에픽 하위로 묻으면 백로그에서 안 보인다.
- **보드는 통합 1개가 정본**, 팀별 보드는 자기 이슈만 보는 뷰. 규칙 딱 하나 — **"스프린트는 한 곳에서만 시작한다"**. *왜*: 스프린트 정보가 프로젝트 전역이라 두 보드가 각자 돌리면 같은 티켓이 이중 배정돼 꼬인다.
- **보드에 무엇을 올릴지 = 라벨로 스코프 지정**(에픽 기준은 계속 늘어 필터가 부서지고, 다른 필드는 오염·요금제 문제로 탈락). **약점도 같이 쓴다** — 라벨이 빠지면 티켓이 보드에서 실종되므로, 자동 부여 규칙과 "누락 색출 필터"로 보완했다.
- 실행분 한 줄: 통합 보드 신설 + 대상 티켓 **53건에 스코프 라벨 일괄 부여** + 스프린트 이름 규칙 확정(주차 표기는 2~3주 가변이라 폐기, 순번 방식으로 — 처음 한 번만 바꾸면 다음부터 자동 증가).

**9. 깨진 오해 모음 — 이 글의 하이라이트**
"이렇게 생각했는데 아니더라" 구조로. **아래 6개만** 쓴다(나머지는 버림):
1. **에픽 하나가 한 번의 반복이다** → 에픽:스프린트는 N:M.
2. **고객 배포용 버전을 따로 만들어야 한다** → 누적이라 불필요. 최신 검증본 하나 지목이면 이전 것이 다 포함된다.
3. **개발 완료 = 끝** → 개발 완료는 아직 검증 전이다. 이게 '완료'로 집계되면 **통계가 거짓말을 한다**. (실제로 그렇게 잡혀 있었다.)
4. **보류한 일은 완료인가 진행중인가** → 둘 다 아니다. 스프린트에서 빼서 백로그로. 완료로 두면 **안 했는데 한 걸로** 집계된다.
5. **보드 컬럼만 나누면 된다** → 컬럼은 상태를 비추는 **거울**일 뿐. 거울을 늘린다고 상태가 생기지 않는다(컬럼 2개를 원하면 상태 2개가 필요).
6. **일단 다 정리하고 시작하자** → **한 번에 뒤엎지 않는다**. 진행 중인 건은 그대로 마무리하고, 다음 주기부터 새 방식을 적용했다.

- 뒤이어 **정리하다 드러난 것들**을 1~2줄로: 버전 항목이 80개 넘게 난립해 있었고(모듈 버전이 릴리스 버전 자리에 들어와 있었다), 종결 사유에 같은 이름이 중복돼 있었으며, 한 보드에 서로 다른 제품 라인이 섞여 있었다. → *프로세스를 세우는 일의 절반은 쌓인 것을 걷어내는 일이었다* 취지.

**10. 남은 것 / 마무리 인용**
- 아직 남은 과제 한 문단(정착·습관화 등 사실 범위에서).
- 마무리 인용(>): 한 줄. 예시 방향 — "프로세스는 문서로 완성되지 않는다. 열차가 실제로 정시에 떠나봐야 한다." (또는 동급, 과장 없이)

### 빼는 것 (명시 — 넣지 말 것)
- RACI 표, 기대효과 나열, 프로세스 문서 목차 같은 **명세서성 내용**.
- 함정 9개 전부 나열(6개만), Jira 세부 조작 절차, 스프린트 생성 권한·QA 이동 범위 같은 도구 운영 디테일.
- 원본 PDF·메모리·사내 문서 **직접 인용/링크**.

### 톤·형식 규약 (블로그 정체성)
- **평서체(-다)**. 존댓말은 여정기·가이드 글만.
- **과장 없음**: 없는 수치 지어내지 않기, "100%"류 금지.
- **영어·전문 용어 첫 등장 시 우리말 병기**(예: Code Freeze(개발 마감), 매니페스트(구성 목록), de-scope(범위에서 제외)).
- **비유 활용**: 열차 / 선반 / 이름표와 내용물 / 거울. 다만 **과용 금지** — 개념당 한 번.
- 기존 글과 같은 골격(리드 인용 → 배경 → 설계 → 함정 → 결과 → 마무리 인용).
