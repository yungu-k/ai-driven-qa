# 개발 핸드오프 — 판단 노출 장치 2개 표준화 (2026-07-23)

> ## 🗣 쉬운 요약
> - **무엇**: 이미 일부 글에만 있던 **두 가지 장치**를 주요 글의 표준으로 승격합니다 — ①「이 자동화가 보장하지 않는 것」 ②「버린 대안과 그 이유」.
> - **왜**: 요즘 평가받는 건 "AI로 만든 결과물"이 아니라 **그 뒤의 판단**입니다. 이 블로그는 결과 수치는 충분한데(1,100+ / 615건 / 53건), **판단 과정이 결론으로만 압축**돼 있어 잘 안 보입니다.
> - **그래서**: 새 글을 쓰는 게 아니라, **이미 본문에 흩어져 있는 판단을 눈에 띄게 드러냅니다.**

> 🔴 **최우선 제약 — 지어내지 말 것**: 이 작업은 **표현·배치 정리**다. 본문에서 **논리적으로 도출되지 않는 내용은 쓰지 않는다.** 필요해 보이는데 근거가 없으면 **비워두고 "저자 확인 필요"로 보고**한다. 없는 한계를 창작하면 이 작업의 목적(정직한 판단 노출)이 정반대로 무너진다.

## 배경 — 왜 이 두 개인가

블로그에 이미 **모범 사례 2개**가 있다. 그걸 표준으로 올리는 것이 이번 작업이다.

- **모범 ①** `db-index-performance-test`의 **"범위와 한계 — 증명 안 한 것도 보고서에 쓴다"** 섹션. 무엇을 증명했고 **무엇은 증명하지 않았는지**를 명시 → 숫자의 신뢰도가 올라간다.
- **모범 ②** `release-train-scrum`의 보드 스코프 결정. 라벨을 고른 이유가 아니라 **에픽·컴포넌트·팀 필드를 각각 왜 버렸는지**를 적었다 → 도메인 이해가 드러난다.

## 📋 개발 핸드오프

### N1. 「보장하지 않는 것」 블록 — 주요 글에 추가

- **무엇을**: 강한 주장을 하는 글에, 그 자동화·시스템이 **덮지 못하는 범위**를 짧은 블록으로 명시한다. 모범 ①과 같은 성격이되 **짧게**(3~4줄 또는 불릿 2~3개).

- **대상 글과, 본문에서 도출 가능한 경계(예시 — 그대로 쓰라는 게 아니라 근거의 위치)**:

  | 글 | 본문에 이미 있는 근거 |
  | --- | --- |
  | `kiosk-automation-infra` | **27건 manual-required** — 환경 한계로 수동 검증에 남긴 영역이 있다는 사실이 본문에 있음 |
  | `realtime-kiosk-monitoring` | **알람 기반**이라는 구조 — 알람이 오지 않는 형태의 문제는 이 화면에 안 잡힌다(본문의 데이터 흐름에서 도출 가능) |
  | `live-issue-triage` | **"분류 누락 0"은 구조적 보장**이지만, 분류가 *옳은가*는 별개(catch-all 설명에서 도출 가능) |
  | `design-change-detection` | **노드 트리 diff** 기준 — 이 방식이 잡는 변화와 못 잡는 변화의 경계(본문 시행착오에서 도출 가능) |

- **수용 기준**:
  - [ ] 위 4편에 「보장하지 않는 것」 성격의 블록이 있다(제목 문구는 글마다 자연스럽게, 형식 통일할 필요는 없음).
  - [ ] 각 항목이 **본문에서 도출 가능한 것**뿐이다. 도출 불가한데 필요해 보이면 **쓰지 말고 보고**.
  - [ ] 기존 수치·주장·톤 **무변경**. 블록 추가만.
  - [ ] `db-index-performance-test`는 **이미 있으므로 손대지 않는다**(모범 원본 보존).

### N2. 「버린 대안」 드러내기 — 이미 있는 것을 눈에 띄게

- **무엇을**: 상당수 글이 **이미 기각 이유를 품고 있는데** 서술에 묻혀 있다. 이를 **또렷하게** 드러낸다. **새 대안을 만들어내지 않는다.**

- **이미 본문에 있는 기각 서술(확인 후 살릴 것)**:

  | 글 | 이미 있는 기각 |
  | --- | --- |
  | `design-change-detection` | 수정 시각(timestamp) 비교를 버리고 **실질적 노드 diff** 채택 |
  | `realtime-kiosk-monitoring` | 범용 대시보드 툴로 빠르게 띄웠다가 **전용 웹앱으로 재구축** |
  | `api-runner-web-ui` | 데스크톱 앱·셸 GUI를 **번들 크기·배포·유지보수** 이유로 탈락 |
  | `field-service-jira-sync` | 자동화 기능이 **월 실행 한도**라 탈락 → 웹훅 / 이름 매칭 → **id 매칭** 전환 |
  | `tc-design-automation`·`qa-report-automation` | API로 안 되는 영역은 **우회 대신 수동 경계**로 확정 |

- **수용 기준**:
  - [ ] 위 글들에서 기각 서술이 **문장 안에 묻히지 않고 읽히게** 된다(굵게·문장 분리·"왜 버렸나" 한 구절 추가 등 — 방식은 재량).
  - [ ] **새 대안을 창작하지 않는다.** 본문에 근거가 없으면 손대지 않는다.
  - [ ] 사실·수치 무변경, 평서체 유지.

### N3. 두 장치를 편집 정책에 등재

- **무엇을**: 앞으로 쓰는 글에도 이어지도록 `docs/content-style.md`에 짧게 등재한다.
- **담을 내용**:
  1. 강한 주장(수치·자동화 효과)을 하는 글은 **「보장하지 않는 것」을 한 블록** 둔다. 한계를 밝히면 나머지 주장의 신뢰가 올라간다.
  2. 중요한 기술·방식 선택은 **버린 대안과 그 이유**를 함께 적는다. 선택보다 기각이 이해를 드러낸다.
  3. 단 **없는 한계·없는 대안을 지어내지 않는다.** 근거 없으면 적지 않는다.
- **수용 기준**: [ ] `content-style.md`에 위 3항이 추가된다.

## 검증 (개발자 규칙)

- 추가한 문장이 **전부 본문 근거에서 도출되는지** 스스로 대조하고, 도출 못 한 항목은 보고에 명시.
- 수치·기존 주장 무변경 확인(diff 확인).
- 빌드 green, 본문 리터럴 `**`·`<del>` 0, 민감정보 0.
- 완료 보고는 돌린 확인 + 결과로. **"근거 없어 못 쓴 것"도 함께 보고**.
