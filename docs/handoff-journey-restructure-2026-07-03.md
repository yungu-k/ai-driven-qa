# 개발 핸드오프 — 여정기 구조 재정비 (린 허브化) (2026-07-03)

> ## 🗣 쉬운 요약
> - **무엇**: 대표글 "여정기"(`repetition-to-ai-judgment-to-human`)를 **줄거리(arc) 중심으로 슬림하게** 다시 잡습니다.
> - **왜**: 방금 각 Phase에 상세글 링크를 걸었는데(hub-spoke), 각 Phase가 여전히 세부 시행착오까지 전부 나열해 **상세글과 내용이 중복**됩니다. 기둥 글은 전체 그림 + 대표 교훈만 담고, 세부는 링크된 가지 글이 짊어지게 합니다.
> - **그래서**: 각 Phase의 시행착오를 "대표 사고 1개만 남기고 나머지는 링크로 위임", 빈 목차·뼈대 불일치·엔딩 중복을 정리합니다. **존댓말·사실·수치·hub 링크는 그대로.**

> 저자 확정(2026-07-03): **방향 A = 린 허브.** 기둥은 arc, 깊이는 가지.

---

## 지배 원칙 (모든 항목에 우선)

각 Phase는 **배경(왜 이 단계가 왔나) + 대표 사고 1개 + 결과/교훈(줄거리를 잇는 한 줄)** 로 압축한다. **열거식 시행착오 세부는 삭제하고, 이미 걸려 있는 상세글 링크가 그 깊이를 담당**한다. 삭제가 아니라 **위임**이다 — 없애는 게 아니라 가지 글로 옮기는 것(가지 글엔 이미 있음, 확인만).

톤·존댓말·수치·사실관계·hub 링크(전체 여정 표, Phase→상세글, 백링크)는 **건드리지 않는다.**

## 📋 개발 핸드오프

### S1. Phase별 시행착오 — "남길 사고 / 위임할 사고" (핵심 작업)

각 Phase에서 **KEEP만 본문에 남기고, DEFER는 삭제**(내용은 링크된 상세글에 이미 존재). 대표 사고는 그 Phase의 결과로 자연스럽게 흐르게.

| Phase | KEEP (남길 대표 사고/교훈) | DEFER → 상세글로 위임 (본문에서 삭제) |
| --- | --- | --- |
| 1 TC 설계 | "규격 문서 먼저" 방향 + **baseline 오설정(TC 7건 삭제 → baseline=마지막 QA 검증)** + 2막 arc(신규=시안 / 릴리스=diff) | 서식 9단계·Priority 색상·Sheets API 한계·코드→테스터언어·git 한계 → `tc-design-automation`, `diff-driven-tc` |
| 2 Bug Report | 배경(왕복) + 결과("마지막 1건까지 같은 품질") | 라벨코드·버전 캐싱·필드 형식 → `bug-report-automation` |
| 3 QA Report | 결과 교훈("데이터 수집이 아니라 **서식 일관성**") | off-by-one 5회·서식깨짐 6건·페이지네이션 50건 → `qa-report-automation` |
| 4 디자인 변경 | 결과("timestamp 비교보다 **실질적 diff**") | 비-UI 오탐·되돌리기·중첩 누락·실행 주기 → `design-change-detection` |
| 5 Regression | 접근(상태 전이 규칙) + 결과("**상태 전이 규칙**을 명확히") | 다중 이슈 연결·관점 혼동 → `regression-and-status-sync` |
| 6 운영 모니터링 | **더 남긴다(서사 전환점)**: 무대 전환 배경 + 600/615 + **catch-all(분류 누락 0건, 구조적)** + 2막(사후→실시간) + 결과 | 8→4→6 색 수렴 세부·SSOT 용어 일화·알람 식별 비일관 → `live-issue-triage`, `realtime-kiosk-monitoring` |
| 7 AI 팀원 | 배경(질문) + 결과(경계가 협업 구조로 확장) | 5개 항목 나열은 2~3줄로 압축(시리즈 페이지가 담당) |

- **수용 기준**:
  - [ ] Phase 1~5·7의 열거식 시행착오 bullet이 KEEP 표 기준으로 정리됐다(대표 사고만).
  - [ ] Phase 6은 전환점이라 상대적으로 풍부하게 유지하되, 위 DEFER 항목은 뺐다.
  - [ ] DEFER한 내용이 해당 상세글에 실제로 있는지 확인(없으면 삭제하지 말고 보고).
  - [ ] 각 Phase가 링크된 상세글로 자연스럽게 이어진다("자세히는 [글]").

### S2. 빈 "## 목차" (§18)

- **무엇을**: 내용 없는 "## 목차" 헤딩이 실제로 자동 TOC로 렌더되는지 확인. 안 되면 제거하거나 테마의 TOC 방식으로 교체.
- **수용 기준**: [ ] 목차가 실제 링크 목록으로 렌더되거나, 아니면 빈 헤딩이 사라진다.

### S3. Phase 뼈대 통일

- **무엇을**: 현재 Phase 1·6은 "1막/2막", 2~5는 "배경/접근/시행착오/결과", 7은 또 다른 구성 — 혼재. S1로 슬림해지면 **일관된 뼈대**(예: 배경 → 핵심/전환 → 결과)로 맞춘다. Phase 1·6의 "막" 서사가 특별히 효과적이면 유지하되, 나머지와 리듬이 맞게.
- **수용 기준**: [ ] Phase 간 소제목 구조가 일관된 리듬을 갖는다(불필요한 편차 제거).

### S4. 엔딩 압축 + 워딩 수정

- **무엇을**: "지향하는 QA 모습"의 5원칙이 도입부 주제·각 Phase 결과와 겹친다. **강한 것만 남겨 synthesis로 압축**하고, opaque한 문장을 고친다.
- **수용 기준**:
  - [ ] 5원칙 중 **"반복은 AI에게, 판단은 사람에게"**(§245)는 도입부 thesis와 중복 → 제목/도입과 안 겹치게 축약하거나 흡수.
  - [ ] **"QA가 할 일과 조직이 할 일을 구분한다"** 문단(§257-259)의 opaque한 문장을 아래로 교체:
    > (기존) "QA 자동화 복잡도를 높이는 원인이라도, 조직 전체가 합의해야 하는 문제는 QA가 단독으로 밀어붙일 수 없습니다. …"
    > (교체) "내 자동화를 복잡하게 만드는 근본 원인이 조직 차원의 문제일 때 — 예컨대 알람 코드 표준화처럼 여러 부서가 합의해야 하는 일 — QA가 혼자 밀어붙일 수는 없습니다. QA는 그 문제를 데이터로 드러내 공유하고, 결정과 추진은 테크리드에게 넘깁니다."
  - [ ] 나머지 원칙(노하우를 코드에 안 가둔다 / 실패는 규칙으로 / 도구는 채택되어야)은 유지하되 각 Phase 결과와의 중복 표현을 다듬는다.
  - [ ] "다음 여정"(§267) 섹션은 유지.

### 손대지 말 것 (명시)

- **"여정을 관통한 구조 — MCP" 섹션(§40)** — 여정 전체의 thesis(단일 Agent·MCP·규격 문서)라 유지.
